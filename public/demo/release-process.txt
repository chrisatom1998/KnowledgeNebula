Our release process follows a consistent, repeatable workflow designed to minimize
risk and ensure quality.

## Release Workflow

1. **Feature branch**: All work starts on a feature branch following our
   [Git Branching](git-branching.md) conventions.

2. **Code review**: PRs require at least two approvals. Security-sensitive changes
   must go through the [Security Review Process](security-review-process.md).

3. **CI pipeline**: [GitHub Actions](github-actions.md) runs linting, unit tests,
   and [Playwright Testing](playwright-testing.md) for end-to-end coverage.

4. **QA sign-off**: The [QA Checklist](qa-checklist.md) is completed against
   the staging environment.

5. **Deploy**: Follow the [Deploy Guide](deploy-guide.md) for blue-green deployment.
   Use [Canary Deployments](canary-deployments.md) for high-risk changes.

6. **Monitor**: After deploy, watch the [Grafana Dashboards](grafana-dashboards.md)
   and [Prometheus Alerts](prometheus-alerts.md) for anomalies.

7. **Rollback**: If issues are detected, follow [Rollback Strategies](rollback-strategies.md).

## Release Notes
Use the [Release Notes Template](release-notes-template.md) to communicate changes
to stakeholders. Tag the release in git following the [RFC Process](rfc-process.md)
version numbering scheme.

## Schema Migrations
If the release includes database changes, coordinate with the
[Schema Migrations](schema-migrations.md) runbook to ensure backward compatibility
with the previous release.
